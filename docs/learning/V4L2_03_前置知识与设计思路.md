## 2. Linux 基础知识

### 2.1 一切皆文件

Linux 中，摄像头是一个**文件**：`/dev/video0`。

```cpp
// 打开摄像头 = 打开文件
int fd = open("/dev/video0", O_RDWR);

// 控制摄像头 = 对文件执行 ioctl
ioctl(fd, VIDIOC_S_FMT, &fmt);

// 读取图像 = 从文件读数据（mmap 方式更高效）

// 关闭摄像头 = 关闭文件
close(fd);
```

这就是为什么 V4L2 代码里到处都是 `fd`（file descriptor，文件描述符）。

### 2.2 ioctl — 设备控制的万能接口

普通文件用 read/write 就够了，但摄像头需要"设置分辨率""设置帧率""查询能力"等特殊操作。
这些操作通过 `ioctl`（I/O Control）完成：

```cpp
int ioctl(int fd, unsigned long request, void *arg);
//        │       │                      │
//        │       │                      └─ 参数（不同 request 对应不同的结构体）
//        │       └─ 请求码（你想做什么操作）
//        └─ 文件描述符（哪个设备）
```

V4L2 定义了一系列请求码，每个对应一个操作：

```
VIDIOC_QUERYCAP   → "你是什么设备？有什么能力？"
VIDIOC_ENUM_FMT   → "你支持哪些像素格式？"
VIDIOC_S_FMT      → "我要用 640x480 YUYV 格式"
VIDIOC_REQBUFS    → "给我分配 4 个缓冲区"
VIDIOC_STREAMON   → "开始采集！"
VIDIOC_DQBUF      → "给我一帧数据"
VIDIOC_QBUF       → "这帧我用完了，还给你"
VIDIOC_STREAMOFF  → "停止采集"
VIDIOC_S_CTRL     → "把对比度设为 128"
```

**为什么代码中用 xioctl 而不是直接用 ioctl？**

```cpp
// ioctl 可能被信号中断，返回 EINTR 错误
// xioctl 自动重试，避免误判为真正的错误
static int xioctl(int fd, int request, void *arg) {
    int r;
    do { r = ioctl(fd, request, arg); }
    while (r == -1 && errno == EINTR);  // 被信号中断就重试
    return r;
}
```

### 2.3 mmap — 零拷贝的秘密

摄像头每秒产生 30 帧图像，每帧 614400 字节（640×480×2）。
如果每帧都从内核空间拷贝到用户空间，CPU 开销很大。

**mmap（内存映射）** 的原理：把内核空间的缓冲区直接映射到用户空间，
用户程序可以直接访问，**不需要拷贝**。

```
不用 mmap（read 方式）：
  内核缓冲区 ──拷贝──→ 用户缓冲区    （每帧都要拷贝 614KB）

用 mmap：
  内核缓冲区 ←──映射──→ 用户指针      （零拷贝，直接访问）
```

代码中的体现：

```cpp
// 1. 请求内核分配缓冲区
ioctl(fd, VIDIOC_REQBUFS, &req);  // "给我 4 个缓冲区"

// 2. 把每个缓冲区映射到用户空间
for (int i = 0; i < 4; i++) {
    ioctl(fd, VIDIOC_QUERYBUF, &buf);  // "第 i 个缓冲区在哪？多大？"
    buffers[i].start = mmap(NULL, buf.length, ...);  // 映射
}

// 3. 采集时直接通过指针访问图像数据
void *data = buffers[buf.index].start;  // 零拷贝！
```

### 2.4 select — 非阻塞等待

摄像头 30fps 意味着每 33ms 产生一帧。如果用阻塞式 ioctl 等待：

```cpp
// 阻塞方式：如果摄像头出问题，程序会永远卡在这里
ioctl(fd, VIDIOC_DQBUF, &buf);  // 一直等，直到有帧数据
```

用 select 可以设置超时：

```cpp
// 非阻塞方式：最多等 2 秒，超时就返回错误
fd_set fds;
FD_ZERO(&fds);
FD_SET(fd, &fds);

struct timeval tv;
tv.tv_sec = 2;   // 超时 2 秒
tv.tv_usec = 0;

int r = select(fd + 1, &fds, NULL, NULL, &tv);
if (r <= 0) {
    // 超时或出错，不会卡死
    return false;
}
// 有数据了，再去取
ioctl(fd, VIDIOC_DQBUF, &buf);
```

**为什么内窥镜需要这个？**
- 摄像头可能被拔掉、排线松动、驱动崩溃
- 阻塞方式会导致整个程序卡死
- select 方式可以检测到异常并优雅处理（显示错误提示而不是卡死）

## 3. 图像基础知识

### 3.1 像素格式 — 一个像素怎么存储

```
RGB888（每像素 3 字节）：
  像素1: [R][G][B]  像素2: [R][G][B]  像素3: [R][G][B] ...
  每像素 3 字节，640×480 = 921600 字节

YUYV（每像素 2 字节，YUV422）：
  [Y0][U0][Y1][V0]  [Y2][U1][Y3][V1] ...
  │    │   │    │
  │    │   │    └─ V 色度（2个像素共享）
  │    │   └─ 第2个像素的亮度
  │    └─ U 色度（2个像素共享）
  └─ 第1个像素的亮度

  每 2 个像素用 4 字节 → 每像素平均 2 字节
  640×480×2 = 614400 字节（比 RGB 省 1/3）
```

**为什么摄像头输出 YUYV 而不是 RGB？**
- 人眼对亮度（Y）敏感，对色度（U/V）不敏感
- YUYV 让相邻两个像素共享色度信息，数据量减少 1/3
- 摄像头传感器原生输出就是 YUV 格式

**为什么我们要转成 RGB？**
- Qt 的 QImage 需要 RGB 格式来显示
- OpenCV 的大部分算法也基于 BGR/RGB

### 3.2 颜色空间转换 YUYV → RGB

```cpp
// 手动转换（我们代码中的实现）
void yuyvToRgb888(const unsigned char *yuyv, unsigned char *rgb, int w, int h) {
    for (int i = 0; i < w * h / 2; i++) {
        int y0 = yuyv[0], u = yuyv[1], y1 = yuyv[2], v = yuyv[3];
        yuyv += 4;
        // YUV → RGB 公式（ITU-R BT.601 标准）
        // R = 1.164*(Y-16) + 1.596*(V-128)
        // G = 1.164*(Y-16) - 0.392*(U-128) - 0.813*(V-128)
        // B = 1.164*(Y-16) + 2.017*(U-128)
        // ... 用整数运算加速（移位代替除法）
    }
}

// 或者用 OpenCV（更简单，且内部有 NEON 优化）
cv::Mat yuyv_mat(height, width, CV_8UC2, data);
cv::Mat bgr_mat;
cv::cvtColor(yuyv_mat, bgr_mat, cv::COLOR_YUV2BGR_YUYV);
```

### 3.3 缓冲区队列 — 生产者消费者模型

V4L2 的缓冲区管理是一个**环形队列**：

```
        ┌─────────────────────────────────────┐
        │          内核缓冲区队列               │
        │                                     │
        │   [buf0] [buf1] [buf2] [buf3]       │
        │     ↑                    │          │
        │     │    VIDIOC_QBUF     │          │
        │     │   （放回队列）      │          │
        │     │                    ↓          │
        │   用户处理完 ←── VIDIOC_DQBUF       │
        │                 （取出一帧）         │
        └─────────────────────────────────────┘

流程：
1. 初始化：4 个缓冲区全部入队（QBUF）
2. 开始采集（STREAMON）
3. 硬件往队列中的缓冲区写入图像数据
4. 用户取出一帧（DQBUF）→ 处理 → 放回（QBUF）
5. 循环 4，直到停止采集（STREAMOFF）
```

**为什么用 4 个缓冲区而不是 1 个？**
- 1 个缓冲区：用户处理时硬件没地方写新帧 → 丢帧
- 4 个缓冲区：用户处理 buf0 时，硬件可以往 buf1/2/3 写 → 不丢帧
- 类似于"双缓冲"的概念，4 个缓冲区更保险

## 4. C++/Qt 基础知识

### 4.1 为什么封装成类

```cpp
// 不封装（面条代码）：
int fd = open("/dev/video0", O_RDWR);
ioctl(fd, VIDIOC_QUERYCAP, &cap);
ioctl(fd, VIDIOC_S_FMT, &fmt);
// ... 几百行代码混在一起

// 封装成类：
V4l2Camera cam("/dev/video0", 640, 480);
cam.open();    // 内部处理所有初始化
cam.start();   // 开始采集
cam.stop();    // 停止
```

封装的好处：
- **复用**：换个摄像头只需改参数，不用改逻辑
- **隔离**：V4L2 的细节不暴露给 GUI 代码
- **测试**：可以单独测试采集模块

### 4.2 为什么用独立线程

```
如果在主线程（GUI 线程）中采集：

  主线程：[等待帧] → [处理帧] → [更新界面] → [等待帧] → ...
                                                │
                                          界面卡顿！
                                    因为等待帧时界面无法响应

用独立线程：

  采集线程：[等待帧] → [发信号] → [等待帧] → [发信号] → ...
                          │
                          ↓ frameReady 信号
  GUI 线程：  [处理事件] → [收到信号] → [更新界面] → [处理事件] → ...
                                                        │
                                                  界面始终流畅！
```

**QThread 的用法：**

```cpp
class V4l2Camera : public QThread {
    void run() override {
        // 这个函数在新线程中执行
        while (m_running) {
            grabFrame();           // 等待并获取一帧
            emit frameReady(img);  // 发信号给主线程
        }
    }
};

// 主线程中
cam.start();  // 启动新线程，自动调用 run()
```

### 4.3 信号槽 — 线程间安全通信

线程之间不能直接操作对方的数据（会导致竞争条件）。
Qt 的信号槽机制提供了安全的线程间通信：

```cpp
// 采集线程中（V4l2Camera）
signals:
    void frameReady(const QImage &image);  // 声明信号

// 采集线程的 run() 中
emit frameReady(image.copy());  // 发射信号（copy 确保数据安全）

// 主线程中（MainWindow）
connect(camera, &V4l2Camera::frameReady,
        this, &MainWindow::onFrameReady,
        Qt::QueuedConnection);  // 跨线程连接

void MainWindow::onFrameReady(const QImage &image) {
    // 这个函数在主线程中执行，可以安全地更新界面
    m_displayImage = image;
    update();  // 触发重绘
}
```

**Qt::QueuedConnection 的含义：**
- 信号在采集线程中发射
- 槽函数在主线程的事件循环中执行
- Qt 自动处理线程间的数据传递
- 不需要手动加锁

### 4.4 QImage — Qt 的图像容器

```cpp
// 从原始 RGB 数据创建 QImage
QImage image(rgbData,          // 像素数据指针
             640, 480,         // 宽、高
             640 * 3,          // 每行字节数（stride）
             QImage::Format_RGB888);  // 像素格式

// 注意：上面的 QImage 不拥有数据，rgbData 被释放后 image 就无效了
// 所以跨线程传递时必须深拷贝：
emit frameReady(image.copy());  // copy() 创建独立的数据副本
```

## 5. 设计思路完整解析

现在回头看设计思路，应该能理解每个决策了：

### 5.1 "从 cap-v4l2 借鉴"的部分

| 借鉴点 | 为什么需要 |
|--------|-----------|
| VIDIOC_QUERYCAP 能力查询 | 启动时确认设备是摄像头、支持 Streaming，避免运行时崩溃 |
| VIDIOC_TRY_FMT 格式试探 | 先试探再设置，如果硬件不支持会返回最接近的参数，而不是报错 |
| select() 等待 | 带超时保护，摄像头异常时不会卡死程序 |
| 曝光/翻转控制 | 内窥镜必备功能，通过 V4L2 Controls 实现 |

### 5.2 "从 VideoMonitor 借鉴"的部分

| 借鉴点 | 为什么需要 |
|--------|-----------|
| QThread 封装 | 采集不阻塞 GUI，界面保持流畅 |
| 信号槽传递 QImage | 线程安全的数据传递，不需要手动加锁 |
| YUYV → RGB 转换 | 摄像头输出 YUYV，Qt 显示需要 RGB |

### 5.3 "与原 VideoMonitor 的区别"

| 改动 | 原因 |
|------|------|
| JPEG → YUYV | YUYV 无需解码，直接转 RGB 给 OpenCV 处理，减少一步开销 |
| 阻塞 → select | 防止摄像头异常时程序卡死 |
| 无查询 → 有查询 | 启动时自动检测硬件能力，代码更健壮 |
| 硬编码 → 可配置 | 分辨率、设备路径等参数化，方便适配不同硬件 |
