# 08 — 常见问题与排错

## 8.1 摄像头相关

### Q: /dev/video0 不存在

```bash
# 1. 检查驱动是否加载
lsmod | grep ov5640
lsmod | grep stm32_dcmi

# 如果没有，手动加载
modprobe ov5640
modprobe stm32_dcmi

# 2. 检查设备树
cat /proc/device-tree/soc/dcmi@4c006000/status
# 如果不是 "okay"，需要修改设备树

# 3. 检查硬件连接
# - OV5640 排线是否插紧
# - 排线方向是否正确（金手指朝向）
# - 供电是否正常

# 4. 查看内核日志
dmesg | grep -i "ov5640\|dcmi\|camera\|i2c"
# 常见错误：
# "ov5640 1-003c: failed to read chip id" → I2C 通信失败，检查硬件连接
# "ov5640 1-003c: failed to set power" → 电源 GPIO 配置错误
```

### Q: v4l2-ctl 报错 "VIDIOC_S_FMT failed"

```bash
# 查看摄像头实际支持的格式
v4l2-ctl --device=/dev/video0 --list-formats-ext

# 如果不支持 YUYV，尝试其他格式
# 常见的 STM32MP157 + OV5640 支持格式：
# - YUYV (YUV422)
# - NV12 (YUV420 semi-planar)
# - RGB565

# 修改代码中的 pixelformat
fmt.fmt.pix.pixelformat = V4L2_PIX_FMT_NV12;  // 或其他支持的格式
# 同时修改颜色转换函数
```

### Q: 画面是绿色/花屏

```bash
# 原因：像素格式不匹配
# 1. 确认实际输出格式
v4l2-ctl --device=/dev/video0 --get-fmt-video

# 2. 确认代码中的颜色转换与实际格式一致
# YUYV → RGB888 的转换公式是否正确
# 如果实际是 NV12，需要用 NV12→RGB 的转换

# 3. 用 OpenCV 转换（更可靠）
cv::Mat yuyv(height, width, CV_8UC2, data);
cv::Mat bgr;
cv::cvtColor(yuyv, bgr, cv::COLOR_YUV2BGR_YUYV);
```

### Q: 帧率很低（<10fps）

```bash
# 1. 检查实际帧率
v4l2-ctl --device=/dev/video0 --get-parm

# 2. 降低分辨率
# 640x480 通常能到 30fps
# 1280x720 可能只有 15fps
# 1920x1080 可能只有 7fps

# 3. 检查是否有其他进程占用 CPU
top

# 4. 确认 OpenCV 编译时开启了 NEON
# 在 OpenCV 编译日志中搜索 NEON: YES
```

## 8.2 交叉编译相关

### Q: 找不到 Qt5 头文件/库

```bash
# 确认 Qt5 安装路径
ls /opt/qt5_arm/lib/cmake/Qt5/

# CMake 中指定路径
cmake .. -DQt5_DIR=/opt/qt5_arm/lib/cmake/Qt5

# 如果使用 qmake
/opt/qt5_arm/bin/qmake endoscope.pro
```

### Q: 找不到 OpenCV 头文件/库

```bash
# 确认 OpenCV 安装路径
ls /opt/opencv_arm/lib/cmake/opencv4/

# CMake 中指定路径
cmake .. -DOpenCV_DIR=/opt/opencv_arm/lib/cmake/opencv4

# 如果 pkg-config 方式
export PKG_CONFIG_PATH=/opt/opencv_arm/lib/pkgconfig:$PKG_CONFIG_PATH
```

### Q: 链接时报 "undefined reference to ..."

```bash
# 常见原因：
# 1. 缺少库链接
# 在 CMakeLists.txt 中添加缺少的库
target_link_libraries(endoscope_demo ... pthread dl m)

# 2. 库的架构不匹配（x86 vs ARM）
file /opt/opencv_arm/lib/libopencv_core.so
# 应输出 ELF 32-bit LSB ... ARM

# 3. Qt MOC 未处理
# 确认 CMakeLists.txt 中有 set(CMAKE_AUTOMOC ON)
```

### Q: 开发板上运行报 "error while loading shared libraries"

```bash
# 缺少动态库
# 1. 查看缺少哪个库
ldd ./endoscope_demo
# 或
./endoscope_demo
# 错误信息会告诉你缺少哪个 .so

# 2. 拷贝库到开发板
scp /opt/opencv_arm/lib/libopencv_*.so* root@192.168.1.100:/usr/lib/
scp /opt/qt5_arm/lib/libQt5*.so* root@192.168.1.100:/usr/lib/

# 3. 更新动态库缓存
ssh root@192.168.1.100 "ldconfig"

# 4. 或者设置 LD_LIBRARY_PATH
export LD_LIBRARY_PATH=/opt/endoscope/lib:$LD_LIBRARY_PATH
```

## 8.3 Qt 显示相关

### Q: "Could not find the Qt platform plugin linuxfb"

```bash
# 1. 确认 Qt 平台插件存在
ls /opt/qt5_arm/plugins/platforms/
# 应看到 libqlinuxfb.so

# 2. 设置插件路径
export QT_QPA_PLATFORM_PLUGIN_PATH=/opt/qt5_arm/plugins
export QT_QPA_PLATFORM=linuxfb:fb=/dev/fb0

# 3. 或者拷贝插件到开发板
mkdir -p /usr/lib/qt5/plugins/platforms/
cp /opt/qt5_arm/plugins/platforms/libqlinuxfb.so /usr/lib/qt5/plugins/platforms/
```

### Q: Qt 程序启动但屏幕无显示

```bash
# 1. 确认 framebuffer 可用
cat /dev/urandom > /dev/fb0  # 屏幕应显示花屏
# Ctrl+C 停止

# 2. 检查 fb0 分辨率
cat /sys/class/graphics/fb0/virtual_size
cat /sys/class/graphics/fb0/bits_per_pixel

# 3. 尝试指定 fb 参数
export QT_QPA_PLATFORM="linuxfb:fb=/dev/fb0:size=800x480:mmsize=154x86"

# 4. 如果有 Wayland/Weston 在运行，需要先停止
systemctl stop weston
```

### Q: 触摸屏不响应

```bash
# 1. 确认触摸设备
ls /dev/input/event*
cat /proc/bus/input/devices

# 2. 测试触摸事件
evtest /dev/input/event0

# 3. 设置 Qt 触摸参数
export QT_QPA_EVDEV_TOUCHSCREEN_PARAMETERS=/dev/input/event0:rotate=0
```

## 8.4 图像处理相关

### Q: CLAHE 处理后颜色异常

```
原因：输入图像的颜色空间不对
解决：确认 QImage → cv::Mat 转换时的颜色空间

// QImage::Format_RGB888 → cv::Mat 是 RGB 顺序
// OpenCV 默认是 BGR 顺序
// 需要转换
cv::cvtColor(mat, mat, cv::COLOR_RGB2BGR);
// 处理完后再转回
cv::cvtColor(mat, mat, cv::COLOR_BGR2RGB);
```

### Q: 去雾算法太慢

```
解决方案（按优先级）：
1. 降采样处理：缩小到一半处理，再放大
2. 减小 radius 参数：从 15 改为 7
3. 隔帧处理：每2帧处理一次
4. 用更简单的去雾方法：直方图拉伸代替暗通道先验
```

### Q: 畸变校正无效果

```
原因：
1. 标定文件未正确加载 → 检查 camera_calib.yml 路径
2. 标定数据不准确 → 重新标定，确保 RMS 误差 < 1.0
3. 标定时的分辨率与运行时不一致 → 标定和运行使用相同分辨率
```

## 8.5 性能优化备忘

### 如果帧率不达标

```
优化优先级（从易到难）：

1. 降低采集分辨率（640x480 → 320x240）
2. 减少同时开启的算法数量
3. 图像处理降采样（处理小图，显示大图）
4. 使用更轻量的算法替代
   - 双边滤波 → 高斯滤波
   - 暗通道去雾 → 直方图拉伸
5. 隔帧处理
6. 多线程：采集线程 + 处理线程 + 显示线程（三级流水线）
7. 使用 OpenCV 的 UMat（如果有 GPU 支持）
```

### 三级流水线架构（进阶优化）

```
线程1（采集）: 帧1采集 → 帧2采集 → 帧3采集 → ...
线程2（处理）:           帧1处理 → 帧2处理 → 帧3处理 → ...
线程3（显示）:                     帧1显示 → 帧2显示 → 帧3显示 → ...

用 QMutex + 双缓冲实现线程间数据传递
```

## 8.6 调试技巧

```bash
# 1. 远程 GDB 调试
# 宿主机
gdb-multiarch ./endoscope_demo
target remote 192.168.1.100:1234

# 开发板
gdbserver :1234 ./endoscope_demo

# 2. 核心转储分析
# 开发板上开启 core dump
ulimit -c unlimited
echo "/tmp/core.%e.%p" > /proc/sys/kernel/core_pattern
# 崩溃后分析
gdb-multiarch ./endoscope_demo /tmp/core.endoscope_demo.xxx

# 3. strace 跟踪系统调用
strace -e trace=ioctl ./endoscope_demo 2>&1 | head -100

# 4. valgrind 检查内存泄漏（如果开发板上有）
valgrind --leak-check=full ./endoscope_demo
```
