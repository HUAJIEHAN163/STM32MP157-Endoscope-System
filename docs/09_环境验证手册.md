# 09 — 环境验证手册

> 野火提供的 Ubuntu 镜像通常已预装了大量开发工具。本文档帮你逐项手动确认，避免重复安装，也避免遗漏。

## 两个环境说明

```
┌─────────────────────────────┐          ┌─────────────────────────────┐
│  宿主机（你的电脑/虚拟机）     │          │  开发板（STM32MP157）         │
│  Ubuntu 18.04/20.04          │   SSH    │  野火提供的 Linux 系统        │
│                              │ ──────→  │                             │
│  职责：                       │  SCP     │  职责：                      │
│  - 编写代码                   │  NFS     │  - 运行编译好的程序           │
│  - 交叉编译（生成ARM程序）     │          │  - 连接摄像头和屏幕           │
│  - 镜头标定工具运行            │          │                             │
└─────────────────────────────┘          └─────────────────────────────┘
```

**核心原则**：代码编写和编译都在宿主机上完成，开发板只负责运行。

## 1.2 交叉编译工具链

这是宿主机上最关键的东西——用来在 x86 电脑上编译出 ARM 程序。

```bash
# 方式一：检查野火 SDK 是否已安装
ls /opt/st/stm32mp1/
# 如果这个目录存在，说明 SDK 已安装
ls: 无法访问'/opt/st/stm32mp1/': 没有那个文件或目录

# 加载 SDK 环境
source /opt/st/stm32mp1/environment-setup-cortexa7t2hf-neon-vfpv4-ostl-linux-gnueabi
# 注意：这个文件名可能略有不同，用 tab 补全或 ls /opt/st/stm32mp1/environment-setup* 查看

# 验证交叉编译器
echo $CC
# 期望输出类似：arm-ostl-linux-gnueabi-gcc -mfpu=neon-vfpv4 ...

$CC --version
# 期望输出 ARM gcc 版本信息

# 验证 sysroot
echo $SDKTARGETSYSROOT
# 期望输出类似：/opt/st/stm32mp1/sysroots/cortexa7t2hf-neon-vfpv4-ostl-linux-gnueabi
ls $SDKTARGETSYSROOT/usr/lib/
# 应该能看到大量 .so 文件
```

```bash
# 方式二：如果没有 SDK，检查是否有独立的交叉编译器
which arm-linux-gnueabihf-gcc
arm-linux-gnueabihf-gcc --version
# 如果存在，记下路径
```

```bash
# 方式三：野火镜像可能把工具链放在其他位置
# 搜索一下
find /opt -name "arm-*-gcc" 2>/dev/null
find /usr -name "arm-*-gcc" 2>/dev/null
```

**结论记录**：
- [ ] 交叉编译器路径：_______________
- [ ] 加载环境的命令：_______________

## 1.3 交叉编译版 Qt5（在宿主机 sysroot 中）

Qt5 是用来交叉编译 GUI 程序的，它应该在宿主机的 SDK/sysroot 中。

```bash
# 先 source SDK 环境（如果还没有）
source /opt/st/stm32mp1/environment-setup-*

# 检查 qmake 是否可用
which qmake
qmake --version
# 期望输出 Qt 5.x.x，并且路径指向 ARM sysroot 中的 qmake
# 如果输出的是 /usr/bin/qmake，那是宿主机本地的 Qt，不是交叉编译版

# 检查 Qt5 cmake 模块
find $SDKTARGETSYSROOT -name "Qt5Config.cmake" 2>/dev/null
# 或
find /opt -name "Qt5Config.cmake" 2>/dev/null
# 期望找到类似：/opt/st/.../lib/cmake/Qt5/Qt5Config.cmake

# 检查 Qt5 关键模块
find $SDKTARGETSYSROOT -name "libQt5Core.so*" 2>/dev/null
find $SDKTARGETSYSROOT -name "libQt5Gui.so*" 2>/dev/null
find $SDKTARGETSYSROOT -name "libQt5Widgets.so*" 2>/dev/null
# 三个都应该找到
```

**结论记录**：
- [ ] 交叉编译版 qmake 路径：_______________
- [ ] Qt5Config.cmake 路径：_______________

## 1.4 交叉编译版 OpenCV（在宿主机 sysroot 中）

```bash
# 先 source SDK 环境
source /opt/st/stm32mp1/environment-setup-*

# 检查 sysroot 中是否有 OpenCV
find $SDKTARGETSYSROOT -name "libopencv_core*" 2>/dev/null
# 期望找到 libopencv_core.so

# 检查 OpenCV cmake 模块
find $SDKTARGETSYSROOT -name "OpenCVConfig.cmake" 2>/dev/null
# 或
find /opt -name "OpenCVConfig.cmake" 2>/dev/null

# 检查 OpenCV 头文件
find $SDKTARGETSYSROOT -name "opencv2" -type d 2>/dev/null
# 期望找到类似：.../usr/include/opencv4/opencv2

# 检查 OpenCV 版本
find $SDKTARGETSYSROOT -name "opencv_version" 2>/dev/null
# 或查看 pkg-config
export PKG_CONFIG_SYSROOT_DIR=$SDKTARGETSYSROOT
pkg-config --modversion opencv4 2>/dev/null || pkg-config --modversion opencv 2>/dev/null
```

**如果 sysroot 中没有 OpenCV**，你需要自己交叉编译，参考 `01_开发环境搭建.md` 的 1.3 节。

**结论记录**：
- [ ] OpenCV 库路径：_______________
- [ ] OpenCV 版本：_______________
- [ ] 是否需要自己编译 OpenCV：是 / 否

## 1.5 宿主机本地 OpenCV（用于标定工具）

标定工具是在宿主机上运行的（不需要交叉编译），所以需要宿主机本地的 OpenCV。

```bash
# 检查本地 OpenCV
pkg-config --modversion opencv4 2>/dev/null || pkg-config --modversion opencv 2>/dev/null

# 或
python3 -c "import cv2; print(cv2.__version__)" 2>/dev/null
```

如果没有：
```bash
sudo apt install libopencv-dev
```

## 1.9 交叉编译验证（最终确认）

写一个最简单的程序，确认整个交叉编译链路通畅：

```bash
# source 环境
source /opt/st/stm32mp1/environment-setup-*

# 写一个测试程序
cat > /tmp/test_cross.cpp << 'EOF'
#include <cstdio>
int main() {
    printf("Hello from STM32MP157!\n");
    return 0;
}
EOF

# 交叉编译
$CXX /tmp/test_cross.cpp -o /tmp/test_cross

# 确认是 ARM 程序
file /tmp/test_cross
# 期望输出：ELF 32-bit LSB executable, ARM, ...
# 如果输出 x86-64，说明用的不是交叉编译器
```

这个编译出来的 test_cross 先留着，等开发板环境确认后传过去运行。

# Part 2：开发板验证

> 以下操作需要先把开发板连上，通过串口或 SSH 登录。

## 2.1 连接开发板

```bash
# 在宿主机上，通过串口连接
sudo minicom -D /dev/ttyUSB0 -b 115200
# 或
sudo screen /dev/ttyUSB0 115200

# 开发板上电，等待启动完成
# 登录（通常 root，无密码或密码 root）
```

---

## 2.2 系统基本信息

```bash
# 内核版本
uname -a
# 期望：Linux ... armv7l（确认是 ARM 系统）

# 系统版本
cat /etc/os-release

# CPU 信息
cat /proc/cpuinfo
# 期望看到：Cortex-A7, neon-vfpv4

# 内存
free -m
# 记录总内存大小

# 存储空间
df -h
# 确认有足够空间存放程序和录像
```

---

## 2.3 网络连通性

```bash
# 开发板上查看 IP
ifconfig
# 或
ip addr

# 如果没有 IP，配置网络
udhcpc -i eth0
# 或
dhclient eth0

# 从开发板 ping 宿主机
ping 192.168.x.x  # 宿主机 IP
# 确认能通
```

然后在宿主机上测试反向连接：
```bash
# 在宿主机上
ssh root@<开发板IP>
# 能登录说明 SSH 通了

# 测试文件传输
scp /tmp/test_cross root@<开发板IP>:/tmp/
# 把之前交叉编译的测试程序传过去
```

---

## 2.4 运行交叉编译的测试程序

```bash
# 在开发板上
chmod +x /tmp/test_cross
/tmp/test_cross
# 期望输出：Hello from STM32MP157!

# 如果报错 "No such file or directory"（文件明明存在）
# 通常是动态链接器路径不对
file /tmp/test_cross
ldd /tmp/test_cross
# 检查依赖的 .so 是否都在
```

## 2.5 摄像头（OV5640）

```bash
# 检查 video 设备
ls /dev/video*
# 期望看到 /dev/video0（或其他编号）

# 如果没有，检查驱动
lsmod | grep ov5640
lsmod | grep dcmi
# 如果没有加载
modprobe ov5640
modprobe stm32_dcmi
# 再检查 ls /dev/video*

# 查看内核日志中的摄像头信息
dmesg | grep -i "ov5640\|dcmi\|camera"
# 期望看到类似：ov5640 ... Product ID 56 40

# 检查 v4l2-ctl 是否可用
which v4l2-ctl
v4l2-ctl --version

# 如果没有 v4l2-ctl
# 野火镜像可能已经有了，也可能需要安装
# apt install v4l-utils  （如果是 Debian/Ubuntu 根文件系统）

# 查看摄像头详细信息
v4l2-ctl --device=/dev/video0 --all

# 列出支持的格式和分辨率（重要！）
v4l2-ctl --device=/dev/video0 --list-formats-ext
# 记录下支持的格式，后面写代码要用

# 抓一帧测试
v4l2-ctl --device=/dev/video0 \
    --set-fmt-video=width=640,height=480,pixelformat=YUYV \
    --stream-mmap=4 \
    --stream-count=1 \
    --stream-to=/tmp/test_frame.raw

ls -la /tmp/test_frame.raw
# 640*480*2 = 614400 字节（YUYV格式）
```

**结论记录**：
- [ ] video 设备路径：_______________
- [ ] 支持的像素格式：_______________
- [ ] 支持的分辨率：_______________

## 2.6 LCD 屏幕

```bash
# 检查 framebuffer
ls /dev/fb*
# 期望看到 /dev/fb0

# 查看屏幕分辨率
cat /sys/class/graphics/fb0/virtual_size
# 期望输出类似：800,480

# 查看色深
cat /sys/class/graphics/fb0/bits_per_pixel
# 通常是 16 或 32

# 测试屏幕是否工作
cat /dev/urandom > /dev/fb0
# 屏幕应该显示随机色块（花屏）
# Ctrl+C 停止
```

**结论记录**：

## 2.7 开发板上的 Qt 运行环境

开发板上不需要 Qt 开发工具，但需要 Qt 的运行时库（.so 文件）。

```bash
# 检查 Qt 库是否存在
find / -name "libQt5Core.so*" 2>/dev/null
find / -name "libQt5Gui.so*" 2>/dev/null
find / -name "libQt5Widgets.so*" 2>/dev/null
# 三个都应该找到

# 检查 Qt 平台插件
find / -name "libqlinuxfb.so" 2>/dev/null
# 或
find / -name "libqeglfs.so" 2>/dev/null
# 至少有一个

# 如果有 Qt 自带的示例程序，可以测试
find / -name "analogclock" 2>/dev/null
# 如果找到了，运行测试
export QT_QPA_PLATFORM=linuxfb:fb=/dev/fb0
/path/to/analogclock
# 屏幕上应该显示一个时钟
```

如果开发板上没有 Qt 库，后续需要从宿主机的 sysroot 中拷贝过去。

---

## 2.8 开发板上的 OpenCV 运行时库

```bash
# 检查 OpenCV 库
find / -name "libopencv_core*" 2>/dev/null
find / -name "libopencv_imgproc*" 2>/dev/null
# 期望找到 .so 文件
```

如果没有，后续需要从宿主机交叉编译的 OpenCV 中拷贝 .so 文件到开发板。
