# 01 — 开发环境搭建（野火 STM32MP157 实际版）

> 本文档基于野火提供的 Ubuntu 18.04 虚拟机镜像的实际验证结果编写。
> 宿主机用户名：embedfire

---

## 整体说明

所有开发和编译工作都在**宿主机（虚拟机）**上完成，开发板只负责运行。

```
宿主机（Ubuntu 18.04 虚拟机）
├── 交叉编译器        → 把 C/C++ 代码编译成 ARM 可执行文件
├── Qt5 ARM 版        → 交叉编译带 GUI 界面的程序
├── OpenCV ARM 版     → 交叉编译图像处理功能
├── OpenCV x86 本地版 → 在宿主机上运行镜头标定工具
├── cmake             → 构建系统，管理编译流程
├── dtc               → 编译设备树（配置摄像头硬件时可能用到）
└── minicom           → 通过串口连接开发板调试
```

**作用**：让我们的内窥镜程序能有图形界面（按钮、滑块、图像显示区域等），并且编译出的 GUI 程序能在 ARM 开发板上运行。

## 1.2 已安装环境（验证过程中安装）

### ✅ g++-arm-linux-gnueabihf

**作用**：ARM 交叉编译的 C++ 编译器。野火镜像只预装了 C 编译器（gcc），没有 C++ 编译器（g++）。Qt 和 OpenCV 都是 C++ 项目，必须有这个。

```bash
# 安装命令（已执行）
sudo apt install g++-arm-linux-gnueabihf

# 验证
which arm-linux-gnueabihf-g++
# 输出：/usr/bin/arm-linux-gnueabihf-g++
```

> **踩坑记录**：野火镜像只装了 `gcc-arm-linux-gnueabihf`，没装 `g++-arm-linux-gnueabihf`。
> 交叉编译 OpenCV 时报错 `The CMAKE_CXX_COMPILER: arm-linux-gnueabihf-g++ is not a full path and was not found in the PATH`。
> 安装 `g++-arm-linux-gnueabihf` 后解决。

### ✅ minicom — 2.7.1

**作用**：串口终端工具。通过 USB-TTL 串口线连接开发板的调试串口，查看启动日志、登录开发板 shell。是嵌入式开发的基本调试手段。

```bash
# 安装命令（已执行）
sudo apt install minicom

# 使用方式
sudo minicom -D /dev/ttyUSB0 -b 115200
```

> **备注**：目前实际使用 Windows 主机上的 MobaXterm 通过串口连接开发板，minicom 作为备用方案。

### ✅ device-tree-compiler (dtc) — 1.4.5

**作用**：编译设备树文件（.dts → .dtb）。如果开发板的设备树中没有配置 OV5640 摄像头，需要修改设备树源文件并重新编译。

```bash

# 使用方式
dtc -I dts -O dtb -o output.dtb input.dts
```

### ✅ OpenCV 本地版（x86）— 3.2.0

**作用**：在宿主机上运行镜头标定工具。标定工具需要读取棋盘格照片、计算相机内参和畸变系数，这个过程在宿主机上完成即可，不需要在开发板上运行。

## 1.3 交叉编译 OpenCV for ARM ✅ 已完成

**作用**：让内窥镜程序在开发板上能使用 OpenCV 的图像处理功能（CLAHE 增强、畸变校正、去雾、降噪等）。这是整个项目图像处理的核心依赖。

> 宿主机上 apt 安装的 OpenCV 3.2.0 是 x86 版本，只能在宿主机上用。
> 开发板是 ARM 架构，需要单独交叉编译一份 ARM 版 OpenCV。

### 步骤

```bash
# 1. 下载 OpenCV 源码
cd ~
git clone https://gitee.com/mirrors/opencv.git -b 3.4.16 --depth 1
# （github 太慢，用 gitee 镜像）

# 2. 创建 toolchain 文件（关键！见踩坑记录坑4）
cat > ~/opencv_arm_toolchain.cmake << 'EOF'
set(CMAKE_SYSTEM_NAME Linux)
set(CMAKE_SYSTEM_PROCESSOR arm)

set(CMAKE_C_COMPILER arm-linux-gnueabihf-gcc)
set(CMAKE_CXX_COMPILER arm-linux-gnueabihf-g++)

set(CMAKE_C_FLAGS "-march=armv7-a -mfpu=neon-vfpv4 -mfloat-abi=hard" CACHE STRING "" FORCE)
set(CMAKE_CXX_FLAGS "-march=armv7-a -mfpu=neon-vfpv4 -mfloat-abi=hard" CACHE STRING "" FORCE)

set(CMAKE_FIND_ROOT_PATH /usr/arm-linux-gnueabihf)
set(CMAKE_FIND_ROOT_PATH_MODE_PROGRAM NEVER)
set(CMAKE_FIND_ROOT_PATH_MODE_LIBRARY ONLY)
set(CMAKE_FIND_ROOT_PATH_MODE_INCLUDE ONLY)
EOF

# 3. 创建构建目录
mkdir ~/opencv/build_arm && cd ~/opencv/build_arm

# 4. cmake 配置交叉编译（使用 toolchain 文件）
cmake .. \
    -DCMAKE_BUILD_TYPE=Release \
    -DCMAKE_INSTALL_PREFIX=/opt/opencv_arm \
    -DCMAKE_TOOLCHAIN_FILE=~/opencv_arm_toolchain.cmake \
    -DENABLE_NEON=ON \
    -DENABLE_VFPV3=ON \
    -DBUILD_TESTS=OFF \
    -DBUILD_PERF_TESTS=OFF \
    -DBUILD_EXAMPLES=OFF \
    -DBUILD_opencv_python2=OFF \
    -DBUILD_opencv_python3=OFF \
    -DBUILD_opencv_java=OFF \
    -DWITH_GTK=OFF \
    -DWITH_QT=OFF \
    -DWITH_V4L=ON \
    -DWITH_FFMPEG=OFF \
    -DWITH_OPENCL=OFF \
    -DWITH_IPP=OFF

# 5. 编译（耗时约 20-40 分钟）
make -j$(nproc)

# 6. 安装到指定目录
sudo make install

# 7. 验证
file /opt/opencv_arm/lib/libopencv_core.so
# 期望输出包含：ELF 32-bit LSB shared object, ARM

ls /opt/opencv_arm/lib/libopencv_*.so
# 应看到 core, imgproc, imgcodecs, videoio 等
```

### toolchain 文件参数说明

| 参数 | 作用 |
|------|------|
| `CMAKE_SYSTEM_NAME=Linux` | 告诉 cmake 目标系统是 Linux（触发交叉编译模式） |
| `CMAKE_SYSTEM_PROCESSOR=arm` | 告诉 cmake 目标 CPU 是 ARM（让 OpenCV 走 ARM 优化分支） |
| `CMAKE_C/CXX_COMPILER` | 指定使用 ARM 交叉编译器 |
| `CMAKE_C/CXX_FLAGS` | ARM 架构编译选项（-march, -mfpu, -mfloat-abi） |
| `CMAKE_FIND_ROOT_PATH` | 交叉编译时查找库和头文件的根路径 |
| `FIND_ROOT_PATH_MODE_PROGRAM=NEVER` | 不在 sysroot 中找可执行程序（用宿主机的） |
| `FIND_ROOT_PATH_MODE_LIBRARY=ONLY` | 只在 sysroot 中找库文件 |

### cmake 参数说明

| 参数 | 作用 |
|------|------|
| `CMAKE_TOOLCHAIN_FILE` | 指定 toolchain 文件，包含所有交叉编译配置 |
| `CMAKE_INSTALL_PREFIX` | 编译产物安装到 `/opt/opencv_arm`，不污染系统目录 |
| `ENABLE_NEON=ON` | 开启 ARM NEON 指令集加速，STM32MP157 的 Cortex-A7 支持 |
| `ENABLE_VFPV3=ON` | 开启硬件浮点加速 |
| `BUILD_TESTS/PERF_TESTS=OFF` | 不编译测试程序，节省时间 |
| `WITH_GTK/QT=OFF` | 不需要 GUI 后端（我们用 Qt 自己显示） |
| `WITH_V4L=ON` | 开启 V4L2 支持（摄像头采集需要） |
| `WITH_FFMPEG=OFF` | 暂不需要视频编解码（如需录像功能后续可开启） |
| `WITH_OPENCL/IPP=OFF` | ARM 平台不支持这些，关掉避免报错 |

### 踩坑记录

**坑1：缺少 C++ 交叉编译器**

```
CMake Error: The CMAKE_CXX_COMPILER: arm-linux-gnueabihf-g++ is not a full path and was not found in the PATH.
```

原因：野火镜像只预装了 `gcc-arm-linux-gnueabihf`，没有 `g++-arm-linux-gnueabihf`。
解决：`sudo apt install g++-arm-linux-gnueabihf`

**坑2：NEON/VFPV3 优化不支持（第一次尝试）**

```
CMake Error at cmake/OpenCVCompilerOptimizations.cmake:535 (message):
  Required baseline optimization is not supported: VFPV3
  (CPU_BASELINE_REQUIRE=;VFPV3;NEON)
```

原因：只指定了 `-DENABLE_NEON=ON -DENABLE_VFPV3=ON`，但没有告诉编译器目标是 ARMv7 架构。
第一次尝试的解决方案：在 cmake 命令行中添加 `-DCMAKE_C/CXX_FLAGS`。
结果：**仍然失败**，见坑4。

**坑3：cmake 失败后重试**

如果 cmake 配置失败，必须先清理再重试：
```bash
rm -rf ~/opencv/build_arm/*
cd ~/opencv/build_arm
# 然后重新执行 cmake 命令
```

不清理直接重跑 cmake 会使用上次的缓存，可能还是报同样的错。

**坑4：命令行指定 CMAKE_C/CXX_FLAGS 和 CPU_BASELINE 仍然失败（最终解决方案）**

即使在 cmake 命令行中加了 `-DCMAKE_C_FLAGS` 和 `-DCMAKE_CXX_FLAGS`，
甚至加了 `-DCPU_BASELINE=NEON -DCPU_DISPATCH=""`，仍然报同样的 VFPV3/NEON 不支持错误。

根本原因：没有设置 `CMAKE_SYSTEM_NAME` 和 `CMAKE_SYSTEM_PROCESSOR`，
cmake 认为目标平台是 x86，走了 x86 的优化检测分支，自然找不到 NEON/VFPV3。

最终解决方案：**使用 toolchain 文件**。在 toolchain 文件中设置：
```cmake
set(CMAKE_SYSTEM_NAME Linux)        # 关键！触发交叉编译模式
set(CMAKE_SYSTEM_PROCESSOR arm)     # 关键！让 OpenCV 走 ARM 分支
```

这两行让 cmake 知道这是交叉编译，目标是 ARM，从而正确进入 ARM 优化检测逻辑。

`-march=armv7-a -mfpu=neon-vfpv4 -mfloat-abi=hard` 这三个 flag 的含义：
- `-march=armv7-a`：目标架构为 ARMv7-A（STM32MP157 的 Cortex-A7）
- `-mfpu=neon-vfpv4`：使用 NEON + VFPv4 浮点单元
- `-mfloat-abi=hard`：使用硬件浮点 ABI（与野火系统一致）

## 1.5 文件传输方式

开发过程中需要频繁把编译好的程序和库文件传到开发板。

### 方式一：scp（推荐，简单可靠）

**作用**：通过网络把文件从宿主机拷贝到开发板。

```bash
# 传输单个文件
scp endoscope_demo root@192.168.0.43:/opt/app/

# 传输 OpenCV 库文件
scp -r /opt/opencv_arm/lib/*.so* root@192.168.0.43:/usr/lib/
```

### 方式二：NFS 共享目录（开发效率最高）

**作用**：宿主机和开发板共享一个目录，宿主机编译完直接就能在开发板上运行，不需要手动拷贝。

```bash
# 宿主机上安装 NFS 服务
sudo apt install nfs-kernel-server

# 创建共享目录
mkdir -p ~/nfs_share

# 配置导出
echo "/home/embedfire/nfs_share *(rw,sync,no_root_squash)" | sudo tee -a /etc/exports
sudo exportfs -arv
sudo systemctl restart nfs-kernel-server

# 开发板上挂载
mount -t nfs 192.168.0.44:/home/embedfire/nfs_share /mnt/nfs -o nolock
```

### 方式三：SD 卡拷贝

**作用**：最简单但效率低，适合偶尔传大文件。把 SD 卡插到宿主机拷贝文件，再插回开发板。

### 摄像头问题排查详情

**排查过程：**

1. `ls /dev/video*` → 不存在
2. `dmesg | grep -i "ov5640\|dcmi\|camera"` → 只有 videodev 模块，无 OV5640/DCMI 信息
3. `cat /proc/device-tree/soc/dcmi@4c006000/status` → **`disabled`**
4. `find /proc/device-tree -name "*ov5640*"` → 无结果（设备树中没有 OV5640 节点）
5. `find /lib/modules -name "*dcmi*"` → 无 .ko 文件
6. `zcat /proc/config.gz | grep -i "DCMI\|OV5640"` → **`CONFIG_VIDEO_STM32_DCMI=y`，`CONFIG_VIDEO_OV5640=y`**

**结论：**
- 内核已经编译了 DCMI 和 OV5640 驱动（`=y`，直接编进内核，不是模块）✅
- 但设备树中 DCMI 状态为 `disabled`，且没有 OV5640 节点 ❌
- 需要通过设备树 overlay 启用

### 野火设备树 Overlay 机制

野火开发板使用 **设备树 overlay** 机制管理外设，配置文件为 `/boot/uEnv.txt`。

```
基础设备树：stm32mp157a-basic.dtb
Overlay 目录：/usr/lib/linux-image-4.19.94-stm-r1/overlays/
配置文件：/boot/uEnv.txt
```

在 `uEnv.txt` 中，每个外设对应一个 `.dtbo` overlay 文件，通过注释/取消注释来启用/禁用。

**关键发现**：摄像头 overlay 已经存在，只是被注释掉了：

```bash
# 当前状态（被注释，未启用）
#dtoverlay=/usr/lib/linux-image-4.19.94-stm-r1/overlays/stm-fire-cam.dtbo
```

### 启用摄像头 ✅ 已完成

在 `/boot/uEnv.txt` 中取消 `stm-fire-cam.dtbo` 的注释，重启后 `/dev/video0` 出现。

```bash
# 操作步骤（已执行）
nano /boot/uEnv.txt
```

## 1.10 附录：如果没有野火预配置镜像，还需要做什么？

> 本项目使用了野火提供的宿主机 Ubuntu 虚拟机镜像和开发板鲁班猫 Debian 镜像，
> 这两个镜像帮我们省掉了大量底层工作。本节记录如果从零开始（裸机 + 纯净 Ubuntu），
> 还需要额外完成哪些工作，以及每项工作的难度和耗时估算。

### 总体对比

```
我们实际做的（有野火镜像）：
  环境验证 → 装几个缺的包 → 交叉编译 OpenCV → 改设备树 overlay → 写应用代码
  耗时：~2天环境 + 专注写代码

从零开始（无任何镜像）：
  装系统 → 装工具链 → 编译内核 → 编译设备树 → 制作根文件系统 → 烧录 →
  交叉编译 Qt → 交叉编译 OpenCV → 调驱动 → 写应用代码
  耗时：~1-2周环境 + 写代码
```

### 宿主机侧：额外需要做的

#### 1. 安装纯净 Ubuntu + 基础开发工具

**野火帮我们做了**：预装了 Ubuntu 18.04 + gcc/g++ + git + make 等基础工具。

**从零开始需要**：
```bash
# 下载 Ubuntu 18.04/20.04 ISO，安装到虚拟机
# 然后手动安装所有开发工具
sudo apt install build-essential cmake git wget unzip \
    autoconf automake libtool pkg-config \
    python3 python3-pip gdb-multiarch \
    libncurses5-dev u-boot-tools device-tree-compiler \
    bison flex libssl-dev bc
```

难度：⭐ 简单 | 耗时：1小时

#### 2. 安装交叉编译工具链

**野火帮我们做了**：预装了 `arm-linux-gnueabihf-gcc 7.5.0`。

**从零开始需要**：
```bash
# 方式一：apt 安装（最简单，但版本可能不匹配）
sudo apt install gcc-arm-linux-gnueabihf g++-arm-linux-gnueabihf

# 方式二：下载 Linaro 工具链（指定版本）
wget https://releases.linaro.org/components/toolchain/binaries/7.5-2019.12/arm-linux-gnueabihf/gcc-linaro-7.5.0-2019.12-x86_64_arm-linux-gnueabihf.tar.xz
tar xf gcc-linaro-*.tar.xz -C /opt/
export PATH=/opt/gcc-linaro-7.5.0-2019.12-x86_64_arm-linux-gnueabihf/bin:$PATH

# 方式三：使用 ST 官方 Yocto SDK（最完整，但体积大）
# 需要先用 Yocto 构建 SDK，或从 ST 官网下载预编译版
```

难度：⭐⭐ 简单 | 耗时：30分钟

#### 3. 交叉编译 Qt5

**野火帮我们做了**：预编译了 `/opt/qt-everywhere-src-5.11.3/`（ARM 版 Qt5）。

**从零开始需要**：这是最耗时的一步之一。

```bash
# 1. 下载 Qt 源码（~500MB）
wget https://download.qt.io/archive/qt/5.11/5.11.3/single/qt-everywhere-src-5.11.3.tar.xz
tar xf qt-everywhere-src-5.11.3.tar.xz
cd qt-everywhere-src-5.11.3

# 2. 创建交叉编译 mkspec 配置
cp -r qtbase/mkspecs/linux-arm-gnueabi-g++ qtbase/mkspecs/linux-stm32mp157-g++
# 编辑 qmake.conf，指定交叉编译器路径

# 3. 配置（需要仔细选择模块，关掉不需要的）
./configure \
    -prefix /opt/qt5_arm \
    -xplatform linux-stm32mp157-g++ \
    -opensource -confirm-license -release \
    -no-opengl -linuxfb \
    -nomake examples -nomake tests

# 4. 编译（非常慢，2-4小时）
make -j$(nproc)
make install
```

难度：⭐⭐⭐⭐ 较难（配置选项多，容易出错） | 耗时：3-5小时（含排错）

> 常见问题：缺少依赖库、linuxfb 插件编译失败、触摸屏支持配置等。
> Qt 交叉编译是嵌入式开发中公认的"坑多"环节。

### 开发板侧：额外需要做的

#### 4. 编译 Linux 内核

**野火帮我们做了**：提供了编译好的内核（4.19.94），且已包含 DCMI 和 OV5640 驱动（`=y`）。

**从零开始需要**：

```bash
# 1. 获取内核源码（ST 官方或野火定制版）
git clone https://github.com/STMicroelectronics/linux.git -b v4.19-stm32mp
cd linux

# 2. 配置内核（需要开启 DCMI、OV5640、V4L2 等）
make ARCH=arm CROSS_COMPILE=arm-linux-gnueabihf- stm32mp157_defconfig
make ARCH=arm CROSS_COMPILE=arm-linux-gnueabihf- menuconfig
# 在 menuconfig 中手动开启：
#   Device Drivers → Multimedia → STM32 DCMI
#   Device Drivers → Multimedia → OV5640
#   Device Drivers → Multimedia → V4L2
#   Device Drivers → Graphics → STM32 LTDC (LCD)
#   Device Drivers → Input → Touchscreen

# 3. 编译内核（30-60分钟）
make ARCH=arm CROSS_COMPILE=arm-linux-gnueabihf- zImage dtbs modules -j$(nproc)

# 4. 安装模块
make ARCH=arm CROSS_COMPILE=arm-linux-gnueabihf- modules_install INSTALL_MOD_PATH=<rootfs路径>
```

难度：⭐⭐⭐⭐ 较难（menuconfig 选项极多，选错会导致外设不工作） | 耗时：2-4小时

#### 5. 编写/修改设备树

**野火帮我们做了**：提供了基础设备树 + overlay 机制，摄像头 overlay 已经写好（stm-fire-cam.dtbo），只需取消注释。

**从零开始需要**：手写设备树节点，这需要对硬件原理图非常熟悉。

```dts
/* 需要手写的内容示例 */

/* 1. 配置 DCMI 接口的引脚复用 */
&pinctrl {
    dcmi_pins_a: dcmi-0 {
        pins {
            pinmux = <STM32_PINMUX('A', 4, AF13)>,  /* DCMI_HSYNC */
                     <STM32_PINMUX('B', 7, AF13)>,  /* DCMI_VSYNC */
                     <STM32_PINMUX('A', 6, AF13)>,  /* DCMI_PIXCLK */
                     <STM32_PINMUX('C', 6, AF13)>,  /* DCMI_D0 */
                     /* ... 还有 D1-D7 共8根数据线 */
            bias-disable;
        };
    };
};

/* 2. 配置 I2C 总线上的 OV5640 */
&i2c1 {
    ov5640: camera@3c {
        compatible = "ovti,ov5640";
        reg = <0x3c>;
        clocks = <&clk_ext_camera>;
        clock-names = "xclk";
        powerdown-gpios = <&gpioa 4 GPIO_ACTIVE_HIGH>;
        reset-gpios = <&gpioa 3 GPIO_ACTIVE_LOW>;
        /* ... 还有电源域、端口配置等 */
    };
};

/* 3. 配置 DCMI 控制器 */
&dcmi {
    status = "okay";
    port {
        dcmi_ep: endpoint {
            remote-endpoint = <&ov5640_ep>;
            bus-width = <8>;
            pclk-sample = <1>;
        };
    };
};
```

难度：⭐⭐⭐⭐⭐ 困难（需要看原理图、查芯片手册、理解引脚复用） | 耗时：1-3天（含调试）

> 这是 BSP（板级支持包）工程师的核心工作，应用层软件开发人员通常不需要从零写设备树。
> 但理解设备树的基本结构是加分项。

#### 6. 制作根文件系统（rootfs）

**野火帮我们做了**：提供了完整的 Debian 根文件系统，预装了 Qt5 运行时库、网络工具、包管理器等。

**从零开始需要**：

```bash
# 方式一：Buildroot（轻量，适合嵌入式）
git clone https://github.com/buildroot/buildroot.git
cd buildroot
make stm32mp157_defconfig  # 如果有的话
make menuconfig
# 选择需要的包：Qt5、OpenCV、v4l-utils、网络工具等
make -j$(nproc)
# 输出 rootfs.ext4

# 方式二：Yocto（ST 官方推荐，但学习曲线陡峭）
# 需要配置 layer、recipe，构建时间 2-8 小时

# 方式三：debootstrap（Debian/Ubuntu 根文件系统）
sudo debootstrap --arch=armhf buster rootfs/ http://deb.debian.org/debian
# 然后 chroot 进去安装需要的包
```

难度：⭐⭐⭐⭐ 较难（Buildroot/Yocto 配置复杂） | 耗时：半天到2天

#### 7. 烧录引导程序（U-Boot / TF-A）

**野火帮我们做了**：eMMC 中已烧录好 TF-A → U-Boot → 内核 → 根文件系统的完整启动链。

**从零开始需要**：

```
STM32MP157 启动链：
ROM Code → TF-A (Trusted Firmware) → U-Boot → Linux Kernel → rootfs

需要分别编译和烧录：
1. TF-A（ARM Trusted Firmware）
2. U-Boot（引导程序）
3. 分区表和烧录脚本
4. 使用 STM32CubeProgrammer 通过 USB 烧录
```

难度：⭐⭐⭐⭐⭐ 困难（启动链复杂，烧录步骤多） | 耗时：1-2天

### 结论

野火镜像帮我们省掉了整个 **BSP（板级支持包）** 层面的工作，让我们可以直接专注于应用层开发（V4L2 采集、图像处理、Qt 界面）。

在公司里，这些底层工作通常由以下角色完成：

| 工作 | 负责角色 | 是否是你的岗位职责 |
|------|---------|-------------------|
| 内核编译、设备树、驱动 | BSP 工程师 / 驱动工程师 | 通常不是，但了解是加分项 |
| 根文件系统、烧录 | 系统工程师 / BSP 工程师 | 通常不是 |
| 交叉编译 Qt/OpenCV | 构建工程师 或 自己做 | 偶尔需要 |
| **V4L2 采集、图像处理、Qt 界面** | **嵌入式软件开发（你的岗位）** | **核心职责** |

所以你的 demo 重点应该放在应用层代码质量上，环境搭建能力作为附加展示即可。
